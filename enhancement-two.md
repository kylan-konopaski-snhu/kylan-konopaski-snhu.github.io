# Enhancement Two - Algorithms and Data Structures
The artifact I selected for enhancement two is the same Java-based contact management system that I used in the first enhancement. I originally created it in CS320 to get familiar with JUnit testing and validation, but it was simple and didn’t have the user interface which I added in enhancement one. For this enhancement, I focused on improving the application’s efficiency and user experience by expanding its use of algorithms and data structures.

I selected this artifact because it provides a good opportunity to demonstrate improvements in performance and functionality. After enhancement one, it was only able to search based on ID or displayed all contacts, but for enhancement two I knew I could improve it greatly by adding a more advanced search functionality, coupled with an efficient sorting functionality. Firstly, I implemented the advanced search which allows users to filter based on first name, last name, phone number, address, or even just a general keyword that searches through all parameters. The underlying data structure used to store contacts is a HashMap, which provides constant time O(1) lookup by ID while still allowing iteration over all values for more flexible searches. This structure supports both fast direct access and the ability to perform linear scans when more complex filtering is required. These searches iterate through the collection linearly in O(n) time and check if the search term is found in the specified parameters, ignoring case. I also introduced a sorting functionality that runs in O(n logn) time where the user can sort based on ID, first name, or last name, in either ascending or descending order. I utilized Java’s comparator interface to define the specific comparison logic for each sorting method. The compactor logic works by dynamically selecting the field to compare and then applying Java’s built in sorting mechanism, which uses a variation of merge sort for efficiency. This allows the application to handle sorting in a reliable and optimized way without having to implement a custom algorithm from scratch. These search and sort options have improved the overall functionality of the application while still being efficient and it demonstrates how many different algorithms can work together within a single application.

I originally just had an ID search function in the first enhancement, but nobody wants to search their contacts based on an ID number. It makes much more sense to use a name or keyword, so it was best to implement that and manage the tradeoffs that search has compared to the constant time ID lookup. One challenge I faced was figuring out how to tie in search and sort features the best way with my command line interface and keeping my code modular and maintainable, but I believe I was able to do it effectively.
